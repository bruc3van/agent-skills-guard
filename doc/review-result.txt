三、安装阶段（Install）

级别	问题	位置
🔴	install_skill 与 confirm_skill_installation 双入口、行为分叉：旧的 install_skill 仍在用 install_from_network 走 GitHub Contents API 递归下载 (skill_manager.rs:1701-1769)，没有 staging、没有多工具链接、没有 installed_commit_sha、没有 source_path/linked_tools。命令层仍把它注册为 install_skill (commands/mod.rs:500)。一旦前端误调用就会产生"半安装"记录。要么删除该入口，要么内部直接 delegate 到 prepare + confirm。	skill_manager.rs:537-697, commands/mod.rs:500-511
🔴	prepare_skill_installation 把 local_path = "__cache__:..." 直接写回 DB：如果用户在 prepare 后从未 confirm/cancel 就关闭 App，下次 get_installed_skills 因 skill.installed == false 不会显示，但 scan_local_skills 用 starts_with("__cache__:") 跳过 (skill_manager.rs:1230)，而 cancel_skill_installation 只在用户显式取消时清理。结果是数据库残留指向已被回收 cache 的脏 local_path，再次进入 confirm 就会 copy_dir_recursive(unknown_path, ...) 失败。建议 prepare 把状态放进"transient" 字段或 in-memory map，不污染 DB 主表。	skill_manager.rs:770-778, skill_manager.rs:1024-1053
🟠	copy_dir_recursive 与 copy_directory 两份几乎相同的实现：前者跟踪 counter，后者 windows 错误信息丰富。差异里 copy_dir_recursive 不创建文件父目录（不是问题，因为它自顶向下创建）但 copy_directory 会重复 create_dir_all。维护成本高，且容易让人改了一处忘了改另一处。建议合并。	skill_manager.rs:857-893, skill_manager.rs:2297-2361
🟠	confirm_skill_installation 用同步 I/O，但 cancel_skill_installation 也走 state.skill_manager.lock().await (commands/mod.rs:553-561)；confirm 调用 tokio::task::spawn_blocking + blocking_lock，cancel 在 async 上直接锁。前端如果"用户点确认 → 5 秒内点取消"，cancel 会一直等到 confirm 释放锁，期间没有任何"正在安装"标志阻止用户重复点击。建议加用户层乐观锁（per-skill state）。	commands/mod.rs:529-561
🟠	enforce_installable_report 把 partial_scan 当硬阻挡 (skill_manager.rs:441-455)；allow_partial_scan 来自前端 boolean。问题在于 prepare 阶段不接受 allow_partial_scan 参数，只在 confirm 阶段重扫时再判断 (skill_manager.rs:460-475)。也就是说"准备"阶段如果有 partial scan，UI 拿到的 report 不会触发硬错误，要等到用户点击"确认安装"才报错——交互上很糟糕。建议 prepare 也 enforce，前端再决定是否提示 override。	skill_manager.rs:701-779
🟡	replace_installation_directory 的备份恢复路径 (skill_manager.rs:367-413) 写得严谨，但 rename_with_retry 只重试 6×250ms (skill_manager.rs:2547-2569)。Windows 病毒扫描经常持有句柄 > 2 秒；建议延长或指数退避。	skill_manager.rs:2547-2569
🟡	install_path 来自用户自定义路径时未走 tool_skill_path_is_compatible_with_source 类校验：用户可以选 ~/.claude/skills 作为 install_path，于是 source = .claude 路径，再为 claude-code 创建链接时自指。create_dir_link 已能识别 same location 直接返回 (link_fs.rs:29-31)，但 linked_tools 会错误地把 claude-code 加进去；之后 sync_skill_to_tools 取消勾选会把同一个目录当链接删掉，带走源。	skill_manager.rs:931-1005

四、卸载阶段（Uninstall）
实现：skill_manager.rs:1056 uninstall_skill + skill_manager.rs:1121 uninstall_skill_path。

级别	问题	位置
🔴	uninstall_skill 遍历 local_paths，但其中第一项通常就是 source (~/.agents/skills/<name>)，是个真目录而非链接。is_dir_link 返回 false ⇒ 走 remove_dir_all。这没问题，但删除顺序是 paths 数组顺序：如果数组里 source 不在末尾、其它工具有降级拷贝（参见 §二 link_fs 降级），先删源后删拷贝就会让拷贝变成"孤儿真目录"——逻辑上还是干净的，但若 source 先于链接被删，is_dir_link 在某些 Windows 实现里对悬空链接返回 false（因为 reparse target 不可访问），代码会 fallback 到 remove_dir_all，用 remove_dir_all 跟随 Junction 删除"目标"目录——虽然此时目标已删，但顺序敏感性是隐患。建议显式两遍：第一遍删所有链接（用 read_dir_link_target 确认指回 source），第二遍删 source。	skill_manager.rs:1067-1084
🔴	uninstall_skill_path 用 path == path_to_remove 字符串相等过滤 (skill_manager.rs:1144)。Windows 上路径大小写/分隔符差异会导致 retain 失败，DB 留下已删的死链路径。应该用 normalize_path_for_compare（skill_manager.rs:62）。	skill_manager.rs:1121-1167
🟠	uninstall_skill_path 删除最后一个路径时不删除 DB 记录，只是 installed=false。对于 is_local_only=true、repository_url=local 的技能这是死记录（用户没法从市场页"再安装"它）。应该走 reconcile_stale_skills 的逻辑分支（local-only → db.delete_skill）。	skill_manager.rs:1146-1151, skill_manager.rs:1619-1646
🟠	前端 executeSkillUninstall 对每个 operationSkillId 串行 await（InstalledSkillsPage.tsx:1005）。多 skill 共享路径的 case 下，第一个 skill 已经把目录删了，第二个会进入 "path.exists() == false" 分支，但 DB 里的 local_paths 仍指向已不存在路径，warnings 满天飞。建议在 get_operation_skill_ids 内合并去重，并在 backend 也容忍多 skill 共享路径（看起来 uninstall_skill_path 已经容忍）。	InstalledSkillsPage.tsx:1000-1020
🟡	uninstall_skill 完全用 log::warn! 吞掉个别删除失败，最后总是返回 Ok。用户点了卸载，看到"已卸载"，实际有目录残留。至少把 errors 汇总成 Err 让前端的 toast 改色，或返回结构化"部分失败"结果。	skill_manager.rs:1056-1118

六、横切关注点
级别	问题
🟠	没有事件总线广播文件系统变更：当外部 CLI（claude code、codex）自己创建/删除 skill 目录时，App 只能等下次 get_installed_skills 刷新。结合 §五的"每次都写 DB" 问题，体验抖动明显。可考虑 notify crate 对四个 skills_dir 做监听，节流后触发 invalidate。

🟡	大量 unwrap / expect：get_skills_directory().expect("Failed to get home directory") (skill_manager.rs:319) 在 CI/容器场景会让进程 abort。
🟡	测试覆盖偏向"build_synced_tool_state / refresh_existing_tool_links" 等小函数；端到端的 prepare→confirm→sync→uninstall 没有集成测试。新增链接相关回归（如 §二 sync_skill_to_tools 非原子性）很容易溜过。
